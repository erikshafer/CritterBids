## Goal

Land the **second and third** Operations read models on top of the M7-S2 scaffold: a **lot board** (`LotBoardView` — an upsert view, one row per listing) and a **bid-activity feed** (an append-only feed, one immutable row per bid). Both are **pure-consumer** ADR 014 Path A read models fed off RabbitMQ. The lot board's rows appear at `ListingPublished` and accrue auction state from the Auctions outcome events; the bid-activity feed appends one row per `BidPlaced`. This slice wires the **two new consumer queues** (`operations-auctions-events` and `operations-selling-events`), implements the two views with their handler families, and proves each **end-to-end** with Testcontainers projection tests against a real Postgres. The M7-S2 settlement queue, its handler, the BC module, the `operations` schema, and the `OperationsBcDiscoveryExclusion` pattern already exist — **build on them, do not recreate them**.

W006 §2 (lot board) and W006 §3 (bid-activity feed) froze both field sets before this slice opened; every field already traces to an existing `CritterBids.Contracts.Auctions.*` or `CritterBids.Contracts.Selling.*` payload. Implement those fields — no more, no fewer. If a new design decision surfaces mid-session, stop and flag (see Open Questions) — do not pivot in-session.

## In scope

1. **`LotBoardView`** — a `sealed record` Marten document keyed by `ListingId`, carrying exactly the W006 §2 field set (`ListingId`, `SellerId`, `Title`, `Format`, `StartingBid`, `ReservePrice?`, `BuyItNow?`, `FeePercentage`, `ScheduledCloseAt?`, `CurrentBid?`, `BidCount`, `HammerPrice?`, `WinnerId?`, `PassReason?`, `WithdrawnBy?`, `WithdrawalReason?`, `Status`, `LastUpdatedAt`). A lot-board `Status` enum whose members realise the W006 §2 derivation chain `Draft (ListingPublished) → Open (BiddingOpened) → Sold | Passed | Withdrawn (terminal)` — member spelling is an implementation choice (W006 freezes the derivation, not the names).
2. **Lot-board handler family — two ADR 014 Sub-Option A siblings, one per source BC:**
   - **Selling-source handler** — consumes `ListingPublished` (seeds the row in `Draft`, populating the catalog-shape fields) and `ListingWithdrawn` (terminal `Withdrawn`, populating `WithdrawnBy`/`WithdrawalReason`). Load-and-preserve any downstream auction state already set.
   - **Auctions-source handler** — consumes `BiddingOpened` (`Open` + `ScheduledCloseAt`, set-once `SellerId`/`StartingBid` reconciliation per W006), `BidPlaced` (`CurrentBid` latest/highest + `BidCount`), `ListingSold` (`Sold` + `HammerPrice` + `WinnerId` + set-once `SellerId` + final `BidCount`), `ListingPassed` (`Passed` + `PassReason` + `BidCount`).
   Each `Handle` is a tolerant upsert (load-or-construct by `ListingId`, mutate via record `with`, `Store`); `Status` is derived from which event arrived per W006 §2; set-once fields (`SellerId`) follow the W006 guard. `SellerId` traces to **three** events (`ListingPublished`, `BiddingOpened`, `ListingSold`) under one set-once guard — `ListingSold` must be able to populate it when it is the first carrier (do not drop `SellerId` from the `ListingSold` mapping). `LastUpdatedAt` is set from each event's own `…At` timestamp (`PublishedAt`/`OpenedAt`/`PlacedAt`/`SoldAt`/`PassedAt`/`WithdrawnAt`) with **latest-wins** semantics — an older out-of-order event must not rewind it (per the S2 `LastUpdatedAt` precedent).
3. **Lot-board status-preservation guard** — terminal states (`Sold`/`Passed`/`Withdrawn`) must **not** regress to `Open` on a late-arriving `BidPlaced` (W006 §2), mirroring the spirit of the S2 `PaidOut`-no-regress guard.
4. **Bid-activity feed** — a `sealed record` **append/feed** view keyed by `BidId` carrying exactly the W006 §3 field set (`BidId`, `ListingId`, `BidderId`, `Amount`, `BidCount`, `IsProxy`, `PlacedAt`). One immutable row per `BidPlaced` — **append, not upsert-in-place**; this is the distinct feed shape §6 calls out. An Auctions-source handler appends one row per `BidPlaced`, with **idempotent dedupe on `BidId`** (re-delivery of the same `BidId` is a no-op, not a second row). The feed has no status and no guards.
5. **`AddOperationsModule()` — additive registration** — contribute the `LotBoardView` and bid-activity feed document types (alongside the existing `SettlementQueueView`) via `services.ConfigureMarten()` in the same `operations` schema. **No** new `AddMarten()` call; no saga, no aggregate, no local event-stream registration (documents only — the S2 "pure Path A consumer needs no event-graph registration" learning holds).
6. **`Program.cs` wiring (routing-only)** — add the `operations-auctions-events` and `operations-selling-events` `ListenToRabbitQueue()` consumers; add publish-route additions for the **S3-consumed events only** (`BiddingOpened`, `BidPlaced`, `ListingSold`, `ListingPassed` to the auctions queue; `ListingPublished` to the selling queue; `ListingWithdrawn` per the routing Open Question); `AutoProvision()` declares both new queues. **No upstream BC code change** — these are routing additions to existing message types. Do **not** add publish routes or handlers for the session events (`SessionCreated`/`SessionStarted`/`ListingAttachedToSession`) that ride the auctions queue — those are S5.
7. **Cross-BC discovery exclusions** — extend the existing `OperationsBcDiscoveryExclusion` registration to every additional existing BC test fixture whose in-process host now discovers the new lot-board/bid-activity Operations handlers (the high-fan-out `BidPlaced`/`ListingSold`/`ListingPassed`/`BiddingOpened`/`ListingPublished` events are co-consumed by Listings, Auctions, Relay, and others). Reuse the existing exclusion class — do not author a new one. Exact set per the Open Question, found by a **red full-suite run**.
8. **`CritterBids.Operations.Tests` additions** — reuse the existing Testcontainers + Alba fixture (it already excludes the six foreign BCs and applies `DisableAllExternalWolverineTransports()`). Add:
   - an **end-to-end lot-board projection test** against real Postgres exercising the full `Draft → Open → Sold` lifecycle (`ListingPublished → BiddingOpened → BidPlaced → ListingSold`), plus the `Passed` and `Withdrawn` terminal paths, asserting field population per W006 §2, the set-once `SellerId` guard, and the terminal-does-not-regress-to-`Open` guard on a re-delivered/late `BidPlaced`;
   - a **seed-late / out-of-order preservation test** proving the Selling-source seed (`ListingPublished`) arriving **after** an Auctions event (e.g. `BiddingOpened` or `BidPlaced` first) fills the catalog-shape fields without clobbering already-set auction state (`CurrentBid`, `BidCount`, `ScheduledCloseAt`, `Status`) or regressing `Status` to `Draft` — the ADR 014 load-and-preserve discipline; and that `LastUpdatedAt` does not rewind on the older-timestamped seed;
   - an **end-to-end bid-activity append test** asserting that N `BidPlaced` events produce N rows (sorted by `PlacedAt`, grouped/filterable by `ListingId`) and that a re-delivered `BidId` is a no-op (no duplicate row).

## Conventions to pin or follow

- **Read-model strategy — ADR 014 Path A, Sub-Option A.** One view per logical entity, one sibling handler **per source BC** (the lot board has two: Selling-source and Auctions-source), tolerant upsert, status-preservation guards. `docs/skills/marten-projections/SKILL.md` owns the mechanics; the prompt points.
- **Upsert vs append/feed (milestone §6).** The lot board is an **upsert** view (one row per `ListingId`, mutated in place). The bid-activity feed is an **append/feed** view (one immutable row per `BidId`, never mutated, dedupe-on-key). These are different shapes — do not implement the feed as an upsert.
- **Operations is a pure consumer.** Consumer handlers return `void`/`Task` and write via the injected Marten session — they do **not** return `OutgoingMessages` and make **no** `IMessageBus` calls. `bounded-contexts.md` §Operations "Integration out: None" is enforced structurally; assert it in a test (per the S2 `tracked.Sent` precedent).
- **No-default-scheme constraint.** S3 introduces no authorization attribute and registers no auth scheme; it leaves `Program.cs`'s auth state untouched. Operations has no HTTP endpoints in S3.
- **Module shape** per `docs/skills/adding-bc-module/SKILL.md`: additive `services.ConfigureMarten()` inside `AddOperationsModule()`; no `AddMarten()` inside the module; same `operations` schema as S2.
- **Canonical bootstrap + `AutoApplyTransactions`** stay in `Program.cs`; the module only contributes types and routes.
- **Cross-BC fixture exclusion naming:** reuse the existing `OperationsBcDiscoveryExclusion` per `critter-stack-testing-patterns/SKILL.md`.
- `sealed record`; `IReadOnlyList<T>` for any collection; no "Event" suffix on type names; participant identifiers are `BidderId`, never "paddle".

## Open questions

- **`ListingWithdrawn` queue routing.** `ListingWithdrawn` is a **Selling**-published contract (`CritterBids.Contracts.Selling`), and W006 §2 groups it under the lot board's **Selling family** sibling handler. But milestone §2 lists `ListingWithdrawn` in the **`operations-auctions-events`** queue row, while the **`operations-selling-events`** row lists only `ListingPublished`. The handler-family organisation (Selling-source sibling) is unaffected either way — a queue is transport, the handler grouping is by source BC — but the publish-route target is ambiguous. Default to the milestone §2 literal (route `ListingWithdrawn` to `operations-auctions-events`) since the milestone doc is authoritative for scope; **confirm** at session start and flag the asymmetry (Selling's `ListingPublished` on the selling queue, Selling's `ListingWithdrawn` on the auctions queue) — if it reads as a milestone typo rather than intent, stop and escalate rather than silently re-routing.
- **Discovery-exclusion fixture set.** The lot-board and bid-activity handlers consume high-fan-out events (`BidPlaced`, `ListingSold`, `ListingPassed`, `BiddingOpened`, `ListingPublished`) that several BCs co-consume and that BC fixtures bus-dispatch via tracked `InvokeMessageAndWaitAsync`. Per the S2 learning, a newly globally-discovered Operations consumer can turn those fixtures red with `NoHandlerForEndpointException` under `MultipleHandlerBehavior.Separated`. Find the exact fixture set by a **red full-suite run** and apply the existing `OperationsBcDiscoveryExclusion` to each named fixture — do not under- or over-apply by guesswork. The set is expected to be **larger than S2's** (Obligations ×2 + Settlement); likely Auctions, Listings, Selling, and Relay fixtures at minimum.
- **`BidPlaced` dual-handler fan-out within Operations.** `BidPlaced` feeds **both** the lot board (upsert) and the bid-activity feed (append) — two Operations handlers on one message type in one assembly. Confirm this composes under `MultipleHandlerBehavior.Separated` and that the Operations fixture's own direct `InvokeMessageAndWaitAsync(BidPlaced)` (if used) does not trip the sticky-routing miss the S2 retro documented; if it does, dispatch through the bus/tracked-session path the skill prescribes rather than reshaping the handlers.
- **Lot-board `Status` enum home and member spelling.** W006 §2 fixes the derivation (`Draft → Open → Sold|Passed|Withdrawn`); the enum's namespace/home and exact member spelling are an implementation choice (W006 explicitly leaves enum member names unfrozen). Keep it Operations-internal (no Contracts type), consistent with the S2 `SettlementQueueStatus` placement.
- **Bid-activity feed storage shape.** W006 §3 freezes the per-row fields and the dedupe-on-`BidId` rule, not the storage mechanics. The expected shape is **one append document per `BidId`** in the `operations` schema (filter/sort by `ListingId`/`PlacedAt` at query time). Pagination, capped-list retention, and sort projection are deferred (W006 leaves them open) — do **not** introduce a capped-list container document or a per-listing aggregate without amending W006 and flagging. Record any Marten indexing choice (e.g. on `ListingId`/`PlacedAt`) in the retro.
