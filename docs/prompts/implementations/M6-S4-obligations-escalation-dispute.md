## Goal

Land the Obligations post-sale coordination saga's **failure paths** and close the `add-obligation-lifecycle` change. The `SendDeadlineEscalation` timer — scheduled at saga start in S3 as a routable no-op stub — gains its body: when the ship-by deadline passes with no tracking, it appends **and emits** `DeadlineEscalated` (a fifth `CritterBids.Contracts.Obligations` integration event — see Decisions resolved) and moves the saga to the non-terminal `Escalated` state. Late tracking still recovers: `ProvideTracking` becomes state-tolerant so a seller who ships after escalation rejoins the happy path. The dispute sub-workflow ships: `OpenDispute` and `ResolveDispute` commands + in-process HTTP endpoints emit the frozen `DisputeOpened` / `DisputeResolved` contracts, with `Refund` and `Closed` terminating the saga via `MarkCompleted()` and `Extension` rescheduling a fresh ship-by deadline and returning to awaiting-tracking without terminating. One staff-facing read model lands — the `ObligationsAwaitingDelivery*` todo-list projection (opsx 8.2). The `OperationsObligationsView` escalation/dispute-queue projection (opsx 8.3) is **reframed as M7 Operations-BC work** and deferred out of this change (see Decisions resolved) — the operator queues are an Operations read model built by subscribing to the integration events this slice publishes, not an Obligations-internal projection. This slice checks off OpenSpec `add-obligation-lifecycle` tasks 6.1, 6.2, 7.1–7.4, 8.2, 9.4, 9.5, defers 8.3 to M7, and flips the closing gate 9.6.

S3 proved the happy path end to end (start → reminder → tracking → auto-confirm → fulfilled) and the cancellable timer chain. S4 adds the alternate transitions only: it does not revisit the happy-path code beyond making `ProvideTracking` state-tolerant. With 6.x/7.x/8.2/9.4/9.5 done, 8.3 deferred to M7, and 9.6 green, the `add-obligation-lifecycle` change is complete and ready to archive. The Relay BC (which actually consumes the integration events this saga publishes) is S5–S7.

## In scope

1. **Missed-deadline escalation body** (opsx 6.1) — `Handle(SendDeadlineEscalation)` gains its body: when the saga is still awaiting shipment at the ship-by deadline, append `DeadlineEscalated` and advance to the non-terminal `Escalated` state. The saga stays alive (no `MarkCompleted()`). A no-op guard short-circuits when state has already advanced past awaiting-shipment (tracking arrived first, the timer was cancelled but defense-in-depth still applies). Per `design.md` "one reminder, then non-terminal escalation."
2. **`DeadlineEscalated` integration event** — new sealed record (no "Event" suffix) in `src/CritterBids.Contracts/Obligations/`, the **fifth** Obligations integration event. Appended to the obligation stream **and** emitted via `OutgoingMessages` (the append+emit taxonomy S3 used for `TrackingInfoProvided`); drives the `Escalated` status in the views and the Operations escalation queue. Carries `ObligationId`, `ListingId`, and `EscalatedAt` (matching the existing contract shapes). This is an **ADR 005 additive contract change** — record it as such; the four contracts frozen at M6-S1 omitted it (see Decisions resolved).
3. **Late-tracking recovery** (opsx 6.2) — `ProvideTracking` becomes state-tolerant: a seller who provides tracking from the `Escalated` state recovers to the happy path (emit `TrackingInfoProvided`, schedule `ConfirmDelivery`, advance to `Shipped`). Cancel any still-pending timer defensively. Per `design.md` "late tracking recovers."
4. **`OpenDispute` command + in-process HTTP endpoint** (opsx 7.1) — `[AllowAnonymous]` `[WolverinePost]`; cascades `OpenDispute` to the saga, which appends + emits `DisputeOpened` (frozen contract: string `Reason` ∈ `NonDelivery | ItemCondition | MissedDeadline`, plus `DisputeId`, `RaisedBy`) and advances to `Disputed`. Opening a dispute does **not** terminate the saga.
5. **`ResolveDispute` command + in-process HTTP endpoint** (opsx 7.2) — `[AllowAnonymous]` `[WolverinePost]`; cascades `ResolveDispute`, which appends + emits `DisputeResolved` (frozen contract: string `ResolutionType` ∈ `Refund | Extension | Closed`, plus `DisputeId`).
6. **Dispute terminal vs. non-terminal branching** (opsx 7.3, 7.4) — on `Refund` and `Closed`, call `MarkCompleted()` (terminal). On `Extension`, recompute a fresh `ShipByDeadline`, reschedule the reminder + escalation timers via `bus.ScheduleAsync()` (reusing the S3 start-time scheduling path), and return to awaiting-tracking **without** terminating. Per `design.md` "Extension is the one non-`MarkCompleted()` path."
7. **`ObligationsAwaitingDelivery*` projection** (opsx 8.2) — todo-list / multi-stream view: a row appears on `TrackingInfoProvided` and self-removes on `DeliveryConfirmed`. Registered Inline.
8. **`OperationsObligationsView` projection (opsx 8.3) — DEFERRED to M7.** The escalation/dispute operator queues are an **Operations-BC read model** (M7), built by subscribing to the `DeadlineEscalated` / `DisputeOpened` / `DisputeResolved` integration events this slice publishes — not an Obligations-internal projection. Do **not** build it here. Instead, edit `openspec/changes/add-obligation-lifecycle/tasks.md` to mark 8.3 deferred-to-M7 with a one-line rationale (cross-BC read models belong to the consuming BC; the Obligations BC's job is to publish the events that feed it). Confirm `openspec validate --strict` still passes after the change-scope edit. Narratives 007 (Setting/Moment 1) and 008 (Cast/Moments) name this view as forward-spec, M7 Operations-owned.
9. **Module + route wiring** — register `DeadlineEscalated` and the `ObligationsAwaitingDelivery*` projection in `AddObligationsModule()`; add publish-only `relay-obligations-events` routes for `DeadlineEscalated`, `DisputeOpened`, and `DisputeResolved` in `Program.cs` (publish-only; the Relay/Operations consumers are S5–S7 / M7).
10. **Tests** (opsx 9.4, 9.5) — escalation + late-tracking-recovery test (deadline passes → `DeadlineEscalated` appended + emitted → `Escalated`; then tracking → recovers → fulfilled); dispute tests (`Refund` terminates with `MarkCompleted()`; `Extension` reschedules a fresh deadline and continues; `Closed` terminates). ## Conventions to pin or follow

- Saga hosting + transitions per ADR-022; `docs/skills/wolverine-sagas.md` owns the `Handle` shape, `[SagaIdentityFrom]`, and `MarkCompleted()`. The `Escalated` and `Disputed` states already exist on `ObligationStatus`.
- `bus.ScheduleAsync()` is the **only** justified `IMessageBus` use in a handler (CLAUDE.md). The `Extension` reschedule reuses it; cancellation/reschedule is via `IMessageStore.ScheduledMessages.CancelAsync(ScheduledMessageQuery)` keyed on the scheduled instant + message type, mirroring `AuctionClosingSaga` — **not** the fictional `bus.CancelScheduledAsync` the skill documents.
- Integration events (`DeadlineEscalated`, `DisputeOpened`, `DisputeResolved`) are all appended to the obligation stream **and** returned via `OutgoingMessages` for cross-BC emission, matching the S3 taxonomy for `TrackingInfoProvided`. `DeadlineEscalated` is the new fifth contract (ADR 005 additive); the other two are frozen at M6-S1.
- **Dispute reason/resolution representation:** the frozen contracts use string-valued `Reason` / `ResolutionType` (the `ListingPassed.Reason` precedent); the saga branches terminal-vs-continue by pattern-matching the `ResolutionType` string constants. Do not change the wire contract. The enum-vs-string-match implementation choice is a carried open question (see below).
- `sealed record`; `IReadOnlyList<T>`; no "Event" suffix; no "paddle"; `[AllowAnonymous]` posture holds through M6; deterministic UUID v5 `ObligationId`; endpoints returning `IResult` need an explicit `using Microsoft.AspNetCore.Http;` (S3 lesson).

## Open questions (carried — saga implementation)

- **`wolverine-sagas.md` cancellation API still wrong (carried from S3).** The skill documents a non-existent `bus.CancelScheduledAsync(id)`; the working API is `IMessageStore.ScheduledMessages.CancelAsync(ScheduledMessageQuery)`. The `Extension` reschedule path exercises this again. Follow the `AuctionClosingSaga` precedent and re-record the owed skill correction in the retro if it has not yet been fixed in a doc pass.
- **Same-instant cross-cancel precision (carried from S3).** `ScheduledMessageQuery` filters by time window + message type, not saga id, so two obligations with timers at the same instant could cross-cancel. The `Extension` reschedule reuses this mechanism. Demo/test instants are unique so it is not exercised; re-note the production limitation in the retro.
- **Dispute reason/resolution representation (carried).** The frozen contracts use string-valued `Reason` / `ResolutionType` (the `ListingPassed.Reason` precedent). The saga branches its terminal-vs-continue behavior by pattern-matching the `ResolutionType` string constants. Decide whether to introduce a private internal enum for branch safety or match the strings directly; record the choice in the retro. Do not change the wire contract.
