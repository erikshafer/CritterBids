## Goal

Resume the platform's authentication posture after five-plus milestones of `[AllowAnonymous]` and put a queryable, **auth-gated** staff surface in front of the Operations read models. This slice (a) registers ADR-024's config-bound `StaffToken` default authentication scheme and the single `StaffOnly` authorization policy, fixing the latent no-`DefaultChallengeScheme` runtime trap that makes a naive `[Authorize]` flip 500 instead of 401; (b) authors **read-only** staff query endpoints over every operator read model now in the BC, born carrying `StaffOnly`; (c) **wire-then-gates** the staff mutations — only `ResolveDisputeEndpoint` exists today, so the three command-only mutations (`WithdrawListing`, `CreateSession`, `StartSession`) get owning-BC HTTP endpoints wired **first**, then gated; (d) proves the gate with authorized-vs-unauthorized assertions on a **real Kestrel** host + Testcontainers Postgres (the SignalR hub is the binding reason for real Kestrel); and (e) supersedes the `[AllowAnonymous]`-through-M6 line in `CLAUDE.md`. ADR-024 is **accepted** — it decided the scheme, the policy, the endpoint set, the hub auth, the 401-vs-403 contract, and the test strategy. S6 writes that decision as code; it does not re-open it.

## In scope

1. **`StaffToken` authentication scheme (ADR-024 Option A).** Register a custom config-bound scheme as the **default authenticate and challenge scheme** in `Program.cs`, replacing the bare `AddAuthentication()` call (the `UseAuthentication()`/`UseAuthorization()` pipeline order is unchanged). It reads the credential from the `X-Staff-Token` header for HTTP requests and **explicitly** from the `access_token` query-string parameter for the `/hub/operations` path (ASP.NET does not wire the query string into a custom scheme automatically — only `AddJwtBearer` ships that). On a match against the configured passphrase it issues an authenticated principal carrying the fixed `staff` claim (type/value pinned by ADR-024 as a shared constant, not a magic string per call site); on absence/mismatch it returns no result so the middleware challenges with 401.
2. **Config-bound, validated passphrase.** The staff token is bound from configuration (`OperationsAuth:StaffToken`, via appsettings / user-secrets / environment) — never hard-coded. Per ADR-024 item 4 the configured token must be non-empty: an empty/missing configured token authenticates **no** request (an empty presented credential must not match an empty configured secret), and host startup **fails** outside test/dev when the staff-gated surfaces are enabled and no token is configured — rather than booting with silently inaccessible (or accidentally open) staff surfaces.
3. **`StaffOnly` authorization policy.** Register it in `Program.cs`'s `AddAuthorization`: `RequireAuthenticatedUser()` + the `staff` claim the scheme issues. Single MVP policy; no per-action or per-role policies ship in M7. The policy does not name a scheme — the default scheme covers it.
4. **Read-only staff query endpoints over all six operator views, authored already gated.** A query surface over **every** read model enumerated from `src/CritterBids.Operations/` — at author time: settlement queue, lot board, bid-activity feed, `OperationsObligationsView`, session-activity board, participant-activity board. Confirm the exact set against code; **all six are in scope this slice** — none defers to a later slice or M8. Each endpoint carries `[Authorize(Policy = "StaffOnly")]` from birth (never `[AllowAnonymous]` then flipped), and each must actually return its view's rows (not a stubbed 200). The *external HTTP contract* — one-endpoint-per-view vs a consolidated dashboard endpoint, route naming, response shape, and any pagination/filtering posture — is an **open question the implementation session flags and stops on if it is not already settled** (it is an external API contract, not an implementation detail to guess; see Open Questions and AUTHORING rule 7).
5. **Queue-state filtering on the obligations reader.** The `OperationsObligationsView` query endpoint(s) must filter to the queue states they serve — the **escalation queue** is `QueueState == Escalated`, the **open-dispute queue** is `QueueState == Disputed`. The S4 tolerant-seed decision means terminal/`Active`/`Resolved`/`Fulfilled` rows can exist for obligations that left (or never entered) an active queue; the reader filters them out (W006 §4 "drop the row from every active queue" = filter, not delete). This is the reader's responsibility, flagged in the S4 retro.
6. **Wire-then-gate the staff mutations.** Per ADR-024 §"Decision" item 3 and the milestone §6 ownership boundary, each mutation stays in its owning BC:
   - `ResolveDisputeEndpoint` (Obligations) **exists** → swap `[AllowAnonymous]` for `[Authorize(Policy = "StaffOnly")]`; success status unchanged (202).
   - `WithdrawListing` (Selling), `CreateSession` and `StartSession` (Auctions) are command-only → **create** the owning-BC `[WolverinePost]` HTTP endpoint **first**, then apply `StaffOnly` to it. Operations does **not** re-home these; the gating attaches to the endpoint S6 introduces inside the owning BC.
7. **`OperationsHub` authorization (Relay) — mandatory this slice.** Apply `[Authorize(Policy = "StaffOnly")]` to `OperationsHub` (`/hub/operations`) so the staff feed is gated; the `StaffToken` scheme's `access_token`-query read (item 1) is what makes the hand-off authenticatable over the WebSocket transport. ADR-024 item 6 **decided** the hub carries `StaffOnly` — gating it is not optional and not deferrable to S7 (a deferral would ship an ungated staff feed). `BiddingHub` stays anonymous (participant-facing). This is a Relay edit, not Operations (ADR-023). The **separate** `OperationsHub` staff-**group**-targeting refinement (M6-S6 standardised to `Clients.All`) is the only deferrable piece and is an **open question** (ADR-024 left it an M7-S1 scope call) — resolve or escalate, do not silently drop it.
8. **Explicit anonymous markers, not implicit.** Per ADR-024 item 5 + milestone §6, once the scheme exists the absence of an attribute is a bug: any participant-facing surface this slice touches keeps an explicit `[AllowAnonymous]`. S6 does **not** sweep the whole codebase — it marks what it touches; the route audit is S7.
9. **Authorized-vs-unauthorized tests on real Kestrel + Testcontainers.** Per ADR-024 item 8: a registered scheme is required for the guarded paths to behave (a no-scheme host 500s rather than 401s), and Alba's in-memory `TestServer` cannot drive a SignalR `HubConnection`, so the hub assertion needs a real socket. Minimum coverage: per guarded HTTP surface, **401** on missing/invalid token and the endpoint's normal **success** status with a valid token (e.g. 202 for `ResolveDisputeEndpoint`, the views' read status for the query endpoints); for `OperationsHub`, connection **rejected** without a valid `access_token` and **established** with one. Whether a **403** path is reachable under the single-shared-secret model is an open question that shapes the matrix. Cross-BC consumer handlers that share an event type apply the existing `*BcDiscoveryExclusion` isolation + `DisableAllExternalWolverineTransports()`; find any newly-red foreign fixtures by a red full-suite run, do not predict them.
10. **`CLAUDE.md` convention edit.** Supersede the `[AllowAnonymous]`-through-M6 bullet (~L131–132) to record that the auth convention resumes at M7-S6 under `StaffOnly` / ADR-024 — staff surfaces gated, participant surfaces explicitly anonymous for MVP. This is the convention-text follow-on ADR-024 records as owed at S6.

## Conventions to pin or follow

- **Auth shape is ADR-024, verbatim.** The scheme is the default authenticate+challenge scheme; the credential is `X-Staff-Token` (HTTP) + `access_token` (the `/hub/operations` query string only — query-string credentials are **not** accepted on non-hub paths); the claim type/value is a shared constant; the policy is the single `StaffOnly`. The staff surfaces are HTTPS-only and the `access_token` parameter is redacted from request logging (ADR-024 consequences). Do not invent a cookie or JWT path — those were the rejected options.
- **401 vs 403 (ADR-024 item 7).** Missing/invalid token → 401 (challenge). An authenticated principal lacking the `staff` claim → 403. In the single-token MVP a valid token always carries the claim, so the observable contrast is 401-vs-success; the 403 path is structural headroom — assert it only if it is actually reachable (open question).
- **Staff-command ownership boundary (milestone §6).** Mutations stay in their owning BC; Operations never re-homes `ResolveDispute`/`WithdrawListing`/the Session commands and never emits another BC's events. The wire-then-gate endpoints are authored in Selling/Auctions, not Operations.
- **`OperationsHub` ownership stays in Relay (ADR-023).** The hub is gated where it lives; Operations builds the read models + query endpoints behind the feed. Any hub edit is a Relay edit.
- **Operations stays a pure consumer.** The query endpoints are read-only over the existing views; they add no `OutgoingMessages`, make no `IMessageBus` call, and append no local stream. `bounded-contexts.md` §Operations "Integration out: None" is unchanged.
- **Test posture is real-Kestrel + Testcontainers (ADR-024 item 8).** Not Alba in-memory `TestServer` for the guarded paths. The `*BcDiscoveryExclusion` + `DisableAllExternalWolverineTransports()` fixture conventions from `critter-stack-testing-patterns` hold; the newly-red foreign-fixture set is found by a red full-suite run, never guessed.
- **Global record conventions.** `sealed record` for any query/response shape; `IReadOnlyList<T>` for collections; no "Event" suffix; participant identifier is `BidderId` (a `string`), never "paddle". Canonical bootstrap + `AutoApplyTransactions` stay in `Program.cs`.

- Narrative 008's operator dispute/escalation surface gains its auth-gated query path and the gated `ResolveDispute` mutation (the rest of S6 is infrastructure with no narrative Moment). No workshop Document-History row is owed.
