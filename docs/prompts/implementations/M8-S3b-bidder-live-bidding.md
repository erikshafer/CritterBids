## Goal

Make the bidder app's listing-detail view a **live bidding surface**: connect it to the `BiddingHub` group for the listing, place a bid through the M8-S3a `POST /api/auctions/bids` endpoint with an **optimistic update that reconciles against the 200 response and rolls back on a 4xx/5xx** (reading the ProblemDetails `reason`), and react to the hub's real-time pushes — the bid feed updating, an outbid indication, an extended-bidding banner when the close is pushed out, and the gavel-fall when the listing sells. This is **narrative 001 Moments 3–7**. The headline architectural deliverable is **ADR-026 — the SignalR integration pattern** (a `SignalRProvider` Context, a `useListen(event, handler)` hook, and the bridge into the TanStack Query cache), authored now that a real hub is wired from the client and the backend's bid + push contracts are lived; the slice also **resolves ADR-013's parked SignalR-integration Deferred Question** to point at ADR-026. **No backend change**: the endpoint and the hub are consumed exactly as M8-S3a / M7 left them.

## In scope

1. **Author ADR-026 — the SignalR integration pattern.** A `SignalRProvider` React Context owning the `HubConnection` lifecycle; a `useListen(event, handler)` hook for components to subscribe to hub events; and the **bridge into the TanStack Query cache** — a push is a `invalidateQueries` / re-query signal, **never** a rendered-as-truth payload (milestone §6 / M7 §5). `Accepted`, alternatives weighed, with a `## Document History` row. **Correct the milestone doc's stale `ADR 014` references to `ADR 026`**, and **amend ADR-013** to flip its SignalR-integration Deferred Question from parked → resolved (→ ADR-026) with a Document-History row. The other ADR-013 Deferred Questions stay parked.
2. **Wire `BiddingHub` live data onto the listing-detail bidding view.** Promote `useBiddingHub` (or wrap it via the new Provider/hook) so the listing-detail view joins the listing's hub group and re-queries `GET /api/listings/{id}` (the lived `CatalogListingView`) on each relevant push — the current high bid, bid count, and high bidder update live.
3. **Bid placement against `POST /api/auctions/bids`.** An amount input on the listing-detail view; submit sends `{ listingId, bidderId: <held ParticipantId>, amount }` (JSON body + `Content-Type: application/json` per LESSONS §A #1). On **200** reconcile the optimistic update against `PlaceBidResponse`; on **4xx/5xx** roll the optimistic update back and surface the ProblemDetails `reason` (e.g. "below the minimum", "exceeds your credit", "listing closed"). Zod-validate the response at the wire boundary.
4. **Real-time beats — Moments 5–7.** The live bid feed (others' bids appearing), an **outbid** indication when the high bidder changes away from the held participant, an **extended-bidding banner** when `ExtendedBiddingTriggered` pushes the close out (reflect the new close time), and the **gavel-fall / sold** state when the listing closes (`ListingSold` / `ListingPassed`). All driven by hub-push → re-query, not by rendering push payloads.
5. **Resolve the two M8-S3a flags frontend-side (no backend change).** (a) **Idempotency:** do **not** auto-retry a bid whose response was lost — roll back the optimistic update and let the user re-submit (a server-generated `BidId` means a blind retry could double-bid). (b) **Concurrent-conflict:** treat **any** non-2xx (including a 5xx from a `DcbConcurrencyException`) as a rollback-and-retry-prompt, the same path as a 4xx rejection. Neither requires a backend change.
6. **Vitest + RTL coverage.** Prove the data layer and the reconciliation: an optimistic update that **rolls back on a mocked rejection** and shows the `reason`; the cache-bridge **re-queries on a simulated hub push**; a 200 response reconciling the optimistic value. (Mocked-fetch tests verify response handling, not request shape — note the live-smoke caveat, LESSONS §D #18; a true browser e2e is M8-S7.)
7. **`docs/retrospectives/M8-S3b-bidder-live-bidding-retrospective.md`** — written last; `**Prompt:**` header + `## Spec delta — landed?` paragraph; records the ADR-026 authoring, the ADR-013 resolution, the milestone ADR-number correction, and how each of the two flags was resolved frontend-side.

## Conventions to pin or follow

- **SignalR client contract:** ADR-023 (`ReceiveMessage` payload, plain-hub broadcast, raw-record payload) + the `wolverine-signalr` skill (`HubConnection`, `.withAutomaticReconnect()`). ADR-026 encodes the *React integration* on top of these; it does not restate the client-library conventions.
- **Relay push = re-query, never render-the-payload:** milestone §6 / M7 §5. The cache bridge invalidates/re-queries the authoritative read endpoint; no push body is rendered as truth. This is the load-bearing rule ADR-026 must encode.
- **Zod at the wire boundary:** the `PlaceBidResponse` and `CatalogListingView` schemas are the single parse points; components consume the parsed types.
- **In-set libraries only:** ADR-013's accepted set (TanStack Router/Query, Zod, Tailwind v4, shadcn/ui, `@microsoft/signalr`, `vite-plugin-pwa`, Vitest + RTL). Any library outside it needs its own ADR — escalate, don't add. (A UI-state library remains an ADR-013-parked question; reach for it only if reconciliation genuinely needs it, and escalate rather than silently adding one.)
- **Wire conventions for the bid POST:** JSON body + `Content-Type: application/json` even though the command is non-empty (LESSONS §A #1); camelCase response keys (LESSONS §A #2); `bidderId` is the held `ParticipantId` Guid from the M8-S2 session bootstrap.
- **Internal-doc prose** (ADR-026, the ADR-013 + milestone amendments, the retro) follows the project's internal-doc conventions; em-dash hygiene is external-prose-only.
