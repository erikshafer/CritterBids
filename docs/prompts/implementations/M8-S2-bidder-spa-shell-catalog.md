**Milestone:** M8 ([React Frontend SPAs](../../milestones/M8-frontend-spas.md)) — slice plan §7, row M8-S2
**Slice:** S2 of M8 (the first *real* bidder-app slice; promotes the M8-S1 connection-proof at `client/bidder/` in place into a routed application shell that renders domain data)
**Narrative:** `docs/narratives/001-bidder-wins-flash-auction.md` (Moment 2 — catalog + listing detail — lands fully; Moment 1's anonymous-session start lands as an *established session*, with the generated display-name header deferred — see Findings) + `docs/narratives/003-bidder-starts-anonymous-session.md` (anonymous-session journey; same display-name deferral)
**Agent:** @PSA (with @UXE consulted on the shell/layout and catalog/detail presentation)
**Estimated scope:** one PR; frontend-only (`client/` tree). No `.cs`, `.csproj`, or `.slnx` file is touched. A routed app shell, catalog + listing-detail views wired through TanStack Query + a Zod wire boundary, the anonymous-session bootstrap, PWA wiring, a Tailwind v4 + shadcn/ui baseline, a routing-library pick recorded as an ADR-013 amendment, and the M8-S2 retro.

## Preconditions

This prompt assumes **M8-S1 has merged**: ADR-013 is `Accepted`, ADR-025 (SPA monorepo layout) is `Accepted`, and the minimal `BiddingHub` connection-proof app exists and builds at `client/bidder/` (Vite 8 + React 19 + TypeScript strict, Tailwind v4 via `@tailwindcss/vite`, `@microsoft/signalr` pinned, `src/useBiddingHub.ts` + `src/App.tsx` rendering a connection-state pill). If any of those is not true, **stop and escalate** — this slice promotes that proof in place; it does not re-establish the foundation. The milestone doc (`docs/milestones/M8-frontend-spas.md`) is authoritative for scope per AUTHORING.md rule 3; this prompt restates the S2 row, it does not widen it.

## Goal

Turn the M8-S1 connection proof into the first real bidder SPA shell: a routed application that establishes an anonymous participant session, renders the public catalog from `GET /api/listings`, and renders a single listing's detail from `GET /api/listings/{id}` — all read-only, all over the existing `[AllowAnonymous]` query surface, with **no backend change whatsoever**. This slice resolves ADR-013's parked **routing-library** Deferred Question — pinned to **TanStack Router** (the first slice with concrete routes) — stands up the Tailwind v4 + shadcn/ui presentation baseline and the PWA wiring ADR-013 commits to "from day one" (deferred off the S1 proof), and introduces the Zod-at-the-wire-boundary + TanStack Query data layer the rest of M8 builds on. It implements **narrative 001 Moment 2** (catalog + listing detail) in full and **establishes** the anonymous session of Moment 1 / narrative 003 — but renders **no live bidding** (that is M8-S3) and **no generated display-name header** (deferred; see Findings). The `BiddingHub` connection from S1 stays connected and observable; no bid feed, no bid placement.

## In scope

1. **Promote the bidder app in place.** Evolve `client/bidder/` from the single-screen connection proof into a routed application shell. Keep the S1 dev-proxy config, the workspace membership, TypeScript strict, and Tailwind-v4-via-`@tailwindcss/vite`. Keep `useBiddingHub` and surface the live connection state somewhere in the shell (a header indicator is fine) — the hub stays **connected and observable**, with no bid data wired.
2. **Resolve the routing-library Deferred Question (ADR-013) → TanStack Router.** The pick is **already made: TanStack Router**, fixed at prompt-authoring time (not a session decision). Rationale to record: it is fully type-safe with first-class search-params-as-state (the catalog's filter/sort and the listing-detail-by-id route want exactly this), and it shares the TanStack lineage with the already-chosen TanStack Query — internal consistency over React Router v7's larger LLM-training footprint, the tradeoff ADR-013 itself names. Wire TanStack Router and record the resolution as an **amendment to ADR-013**: flip the routing Deferred Question from parked to resolved with **TanStack Router** + that one-paragraph reason, and add a `## Document History` row dated this session. The other four Deferred Questions stay parked.
3. **App shell + routes.** A layout shell (header with app identity + the live-connection indicator; a content outlet) and at minimum two routes: the catalog (`/`) and listing detail (`/listing/:id` or the router's idiomatic equivalent). 404/unknown-route handling present.
4. **Anonymous session bootstrap.** On first load (no session held), `POST /api/participants/session`, hold the returned participant **id** in client state for the session's lifetime (the established-session token later slices' bid placement will carry). Render the session as **established/anonymous** — **no generated display-name header this slice** (deferred; see Findings and Out of scope). Handle the POST failing gracefully (the catalog is still anonymously readable).
5. **Catalog + listing-detail data layer.** TanStack Query wired to `GET /api/listings` (list) and `GET /api/listings/{id}` (detail), **read-only**. A **Zod schema mirroring `CatalogListingView`** validates the wire boundary (per ADR-013 — Zod at the boundary); the parsed type is what components consume. Empty catalog renders an empty state (not an error); a missing detail id renders a not-found state (the endpoint 404s). Loading and error states present for both. Treat the data as authoritative-on-fetch; no hub push drives these queries this slice (that bridge is ADR-014 / M8-S3).
6. **Tailwind v4 + shadcn/ui baseline.** Establish the presentation baseline: Tailwind v4 utilities in use, and shadcn/ui initialized with components **copied in only as the catalog/detail/shell need them** (the shadcn copy-in model — no bulk component-catalog scaffold per milestone §3).
7. **PWA wiring (honoring ADR-013 "day one").** Wire `vite-plugin-pwa` — web-app manifest + service-worker registration — now that a real shell exists to install (the S1 retro explicitly carried this forward to "the first real bidder shell"). App-shell precache is sufficient; **offline data scope stays an ADR-013-deferred question** (do not design offline catalog caching or offline bid queueing here).
8. **Vitest baseline + minimal component coverage.** Stand up Vitest + React Testing Library (Vite-native config) and add **minimal** coverage proving the data layer: the Zod schema accepts a representative `CatalogListingView` and rejects a malformed one; the catalog view renders a list and its empty state. A comprehensive component suite and **all Playwright e2e are out of scope** (e2e is M8-S7).
9. **`docs/retrospectives/M8-S2-bidder-spa-shell-catalog-retrospective.md`** — written last; carries the `**Prompt:**` header line and a `## Spec delta — landed?` paragraph, records the routing pick + the user sign-off, and records the display-name deferral as a carry-forward.

- **Settlement-outcome view** (M8-S4), **the ops app / `client/ops/` / `OperationsHub`** (M8-S5/S6), and **the staff-token auth client** (ADR-024 client side) — none ship this slice.
- **shadcn/ui full component-catalog scaffolding**, a **UI-state-management library** (Zustand/TanStack Store — ADR-013-deferred, no concrete need yet), and **`react-hook-form`/Zod form wiring** (no forms this slice; the catalog/detail are read-only).
- **Offline PWA data scope** (caching listings, queueing offline bids) — app-shell precache only; offline data strategy stays an ADR-013-deferred frontend-skill question.
- **The frontend CI job** — M8-S7 housekeeping (`project_frontend_ci_not_wired`); not this PR.

## Conventions to pin or follow

- **SPA layout & dev-server:** ADR-025 — `client/bidder/` is the app's home; the Vite dev-proxy (`/api`, `/hub` → `:5180`, `ws:true`) is the dev reach to the API + hub. Build within it; do not introduce CORS or a second dev mechanism.
- **Stack:** ADR-013 owns the library set. This slice **resolves ADR-013's routing Deferred Question to TanStack Router** (recorded as an ADR-013 amendment) and uses only in-set libraries (TanStack Router, TanStack Query, Zod, Tailwind v4, shadcn/ui, `@microsoft/signalr`, `vite-plugin-pwa`, Vitest + RTL). Any library outside the set requires its own ADR — escalate, don't add.
- **Zod at the wire boundary:** the `CatalogListingView` Zod schema is the single parse point; components consume the parsed type, not raw JSON. (`tanstack-query-best-practices`, `zod` skills.)
- **Relay push = re-query, never render-the-payload:** milestone §6 / M7 §5. Not exercised this slice (no hub data), but the data layer must not be architected to render a push body as truth — keep fetched query state authoritative so the M8-S3 cache-bridge slots in cleanly.
- **SignalR client conventions:** the existing `useBiddingHub` follows `docs/skills/wolverine-signalr/SKILL.md` + ADR-023 (`ReceiveMessage`, `.withAutomaticReconnect()`, raw-record payload). Leave that contract intact; do not invent a client message-type discriminator (that is ADR-014 / M8-S3).
- **Internal-doc prose** (the ADR-013 amendment + the retro) follows the project's internal-doc conventions; em-dash hygiene is external-prose-only and does not apply.

## Open questions

- **Router pick — settled: TanStack Router.** Fixed at prompt-authoring time by the user; no in-session evaluation or sign-off gate remains. The session's only routing task is to wire it and record the ADR-013 amendment (resolved → TanStack Router, with the type-safety + TanStack-lineage rationale). If wiring surfaces a reason to revise any *other* ADR-013 sub-choice, escalate rather than fold it into the routing amendment.
- **Display-name header (deferred, carry-forward).** The anonymous bidder's generated name is not surfaced by any current endpoint and M8 forbids backend changes, so the header is deferred this slice (user-confirmed at authoring). The future slice that renders it will need a backend read path — flag in the retro which slice should own that (and whether it warrants enriching the session response or adding a `GET /api/participants/{id}`) so it is not silently dropped.
- **`CatalogListingView` as the detail shape.** The lived detail endpoint returns the same `CatalogListingView` the list returns (the milestone's surface table optimistically named a `ListingDetailView` that does not exist). The Zod schema and detail view bind to `CatalogListingView`. If the detail view genuinely needs a field the read model lacks, that is a backend concern — escalate; do not invent a client-side shape.
- **Session persistence across reloads.** Whether the established participant id is held in memory only (re-minted on reload) or persisted (e.g. `sessionStorage`) is a UX detail for the session to settle with @UXE; in-memory is an acceptable floor. Re-minting must not duplicate-charge any later flow — but since no bidding ships here, the floor is safe this slice.
