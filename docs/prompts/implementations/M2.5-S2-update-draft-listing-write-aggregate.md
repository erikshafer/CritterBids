## Goal

Apply the explicit `[WriteAggregate]` stream-ID override to `UpdateDraftListingHandler.Handle()`
and add a matching dispatch test that proves the override works end-to-end through Wolverine.

M2.5-S1 resolved the same fragility for `SubmitListing` and flagged this as the remaining
instance — the `UpdateDraftListing` command has `ListingId` on a `SellerListing` aggregate, so
Wolverine's convention looks for `SellerListingId` and dispatch fails. No HTTP endpoint currently
exercises this path, which is precisely why it must be fixed and tested now — before M3 adds
one and rediscovers the same runtime surprise.

M2.5 re-opens as two sessions (S1 + S2) rather than the originally planned S1-of-1, to isolate
this single-deliverable fix into its own reviewable PR per rule 1.

## Conventions to pin or follow

- `nameof(UpdateDraftListing.ListingId)` for the attribute argument — not a string literal.
  The rename-safety rationale is documented in `wolverine-message-handlers.md` under the
  explicit-override subsection; this session applies the rule, it does not restate it.
- Dispatch test fixture: reuse `SellingTestFixture` / `SellingTestCollection` — do not
  create a parallel fixture. `SubmitListingDispatchTests.cs` is the reference shape.
- Seed via `IDocumentSession.Events.StartStream<SellerListing>(listingId, new DraftListingCreated(...))`
  directly — consistent with the S1 dispatch test. Do not go through
  `POST /api/listings/draft` for seeding.
- Assertion pattern: re-aggregate with `querySession.Events.AggregateStreamAsync<SellerListing>(listingId)`
  and assert on the post-update aggregate state, not on projected documents. The projection
  path is not what this test is proving.
