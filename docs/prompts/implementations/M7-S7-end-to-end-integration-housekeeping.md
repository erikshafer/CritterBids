## Goal

Close M7 by proving the Operations BC works end to end and tidying the milestone's loose threads. S2–S5 seeded the six views in isolation against a real Postgres (settlement queue; lot board + bid-activity feed; obligations view; session + participant boards); S6 put a gated query surface in front of all six and proved the auth gate per-surface. S7 is the slice that demonstrates the **whole pipe at once**: a single cross-BC journey that drives **real** integration activity — a settlement, a published lot with bids, an escalated obligation that opens a dispute and is then extended and fulfilled, a Flash Session with an attached listing, and a participant joining — and asserts that **every** operator read model (settlement queue, lot board + bid-activity feed, `OperationsObligationsView` escalation **and** dispute queues, session activity board, participant activity board) reflects that activity, read back through S6's `StaffOnly`-gated query endpoints. Around that proof S7 performs the milestone's housekeeping: a `Program.cs` route audit confirming every `operations-*` consumer `ListenTo` and its publish-route additions are correctly wired and that every M7-touched endpoint carries an explicit authorization attribute; the `bounded-contexts.md` flip of Operations from "Planned" to active; folding the lived S6 hub-gating reality into the `wolverine-signalr` skill; and the milestone close — the S7 session retrospective **and** the M7 milestone retrospective, with the test-count baseline updated. S7 adds **no** new Operations view, handler, consumer route, query endpoint, or contract type — the milestone's behavior is complete at S6; S7 is verification + closure.

## In scope

1. **One cross-BC end-to-end journey test driving real activity into every operator view.** A single integration test (per-view supplemental assertion files are permitted, but the journey itself is one coherent scenario) that produces **genuine** consume-path activity — not a per-view seed — across all five source BCs, and asserts every Operations read model reflects it:
   - **Settlement queue** — a settlement reaches the queue (e.g. `SettlementCompleted`/`SellerPayoutIssued`, with at least one `PaymentFailed`-flagged row to prove the staff-attention path).
   - **Lot board + bid-activity feed** — a `ListingPublished` lot goes `Open` via `BiddingOpened`, takes ≥2 `BidPlaced` (proving the append feed accrues one row per bid), and reaches a terminal outcome (`ListingSold`/`ListingPassed`).
   - **`OperationsObligationsView` escalation + dispute queues** — a `DeadlineEscalated` lands a row on the escalation queue, `DisputeOpened` moves it to the dispute queue, `DisputeResolved { Extension }` returns it to the active set, and `ObligationFulfilled` clears it — the narrative-008 operator arc, asserted through **both** the `obligations/escalations` and `obligations/disputes` gated endpoints.
   - **Session activity board** — `SessionCreated` → `ListingAttachedToSession` → `SessionStarted` produces a session row with the attached listing and `Started` status.
   - **Participant activity board** — `ParticipantSessionStarted` produces a participant row (`BidderId` as `string`).
   The journey **reads back through S6's `StaffOnly`-gated query endpoints** with a valid staff token (proving consume→project→**query** end to end, not just consume→project), on a **real Postgres** via Testcontainers, reusing the S6 `StaffAuthTestFixture` host lineage (the only host that maps every module **and** the gated endpoints **and** carries the auth scheme). ## Conventions to pin or follow

- **Assert on the Operations read model, never on a Relay push (milestone §5 eventual-consistency note).** The Relay `OperationsHub` push and the Operations projection are two independent consumers of the same upstream event and are **unordered**. The journey test asserts state by querying the Operations read model (through the gated endpoints), never by observing a Relay push — and tolerates projection lag with the established tracked-session await (`InvokeMessageAndWaitAsync` for single-handler events; `SendMessageAndWaitAsync` for fanned-out events like `BidPlaced`, per the S3 sticky-routing lesson), not a sleep.
- **Operations stays a pure consumer (milestone §6).** The journey exercises consume→project→query only; it asserts the pure-consumer contract holds end to end (`tracked.Sent` empty for the Operations handlers; no `OutgoingMessages`; no `IMessageBus`). `bounded-contexts.md` §Operations "Integration out: None" is unchanged by the status flip.
- **Empirical discovery-exclusion (S2–S5 rule).** The journey runs on a host that maps every module, so the `*BcDiscoveryExclusion` + `DisableAllExternalWolverineTransports()` posture from `critter-stack-testing-patterns` applies; any newly-red foreign fixture is found by a **red full-suite run**, never predicted. (S4/S5 found the set empty because the foreign exclusions are whole-namespace — expect the same, but measure.)
- **Staff-token read-back uses the S6 contract.** The journey reads the gated endpoints with the `X-Staff-Token` header (HTTP) and, if it asserts the hub at all, the `access_token` query for `/hub/operations`; the obligations surface is the two endpoints (`obligations/escalations` = `QueueState.Escalated`, `obligations/disputes` = `QueueState.Disputed`). No new endpoint, no contract widening.
- **Route audit is verify-then-close, not redesign.** Confirm the §2/§5 enumerated routes and the explicit-attribute rule; close only pure routing/attribute gaps. A discovered behavior gap escalates (AUTHORING rule 7).
- **Global record conventions.** `sealed record` for any test-only DTO/response shape; `IReadOnlyList<T>` for collections; no "Event" suffix; participant identifier is `BidderId` (a `string`), never "paddle". The canonical bootstrap + `AutoApplyTransactions` stay in `Program.cs`.
