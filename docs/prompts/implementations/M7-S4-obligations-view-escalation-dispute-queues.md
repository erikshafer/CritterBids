## Goal

Land the **fourth** Operations read model on top of the M7-S2/S3 scaffold: the **`OperationsObligationsView`** — the narrative-008 operator surface carrying **two derived queues**, the **escalation queue** (from `DeadlineEscalated`) and the **open-dispute queue** (from `DisputeOpened`, cleared by `DisputeResolved`), with rows dropped from the active set on `ObligationFulfilled`. ## In scope

1. **`OperationsObligationsView`** — a `sealed record` Marten document keyed by `ObligationId`, carrying exactly the W006 §4 field set (`ObligationId`, `ListingId`, `DisputeId?`, `RaisedBy?`, `DisputeReason?`, `ResolutionType?`, `ResolutionParticipantId?`, `WinnerId?`, `SellerId?`, `EscalatedAt?`, `DisputeOpenedAt?`, `DisputeResolvedAt?`, `FulfilledAt?`, `QueueState`). A `QueueState` enum whose members realise the W006 §4 derivation `Escalated → Disputed → (Resolved-or-back-to-active) → Fulfilled` — member spelling is an implementation choice (W006 freezes the derivation, not the names). This is **one** view modelling two **derived** operator queues (escalation = `QueueState == Escalated`; open-dispute = `QueueState == Disputed`), **not** two views and **not** a queue-discriminator *row* split — it is a per-obligation upsert with a `QueueState` status (contrast S3: the lot board is an upsert view, the bid feed is an append feed; S4 is upsert-only, shaped like the lot board but with queue membership derived from `QueueState` rather than from a `Status` rank, because one transition runs backward).
2. **One Obligations-source handler family — ADR 014 Sub-Option A, a single sibling class** (Obligations is the only source BC) consuming the four events, each `Handle` a tolerant upsert (load-or-construct by `ObligationId`, mutate via record `with`, `Store`), pure-consumer (returns `Task`, **no** `OutgoingMessages`, **no** `IMessageBus`):
   - **`DeadlineEscalated`** → seed/advance to `QueueState = Escalated`; set `EscalatedAt`; set-once `ListingId`.
   - **`DisputeOpened`** → advance to `QueueState = Disputed`; set `DisputeId`/`RaisedBy`/`DisputeReason` (from `.Reason`)/`DisputeOpenedAt` (from `.OpenedAt`).
   - **`DisputeResolved`** → set `ResolutionType`/`ResolutionParticipantId` (from `.ParticipantId`)/`DisputeResolvedAt` (from `.ResolvedAt`); branch on `ResolutionType`: **`Extension`** returns the row to the **active set** (out of both queues, **not** terminal — the narrative-008 Moment-2 transition, a deliberate move *out of* `Disputed`); **`Refund`**/**`Closed`** resolve **terminally**.
   - **`ObligationFulfilled`** → advance to terminal `QueueState = Fulfilled`; set `WinnerId`/`SellerId`/`FulfilledAt`; drop the row from every active queue.
   Each event's own timestamp populates the matching `…At` field. **Set-once `ListingId`** (the cross-view join key, on all four events) follows the W006/S2 set-once guard. Read W006 §4 field-by-field for set-once-vs-last-write — do **not** assume a uniform guard (the S2 `SellerId`-last-write lesson).
3. **Status-preservation / transition guards** — `Fulfilled` is **terminal** and must not regress (W006 §4); `Refund`/`Closed` terminal resolutions must not regress to `Disputed`/`Escalated` on a re-delivered earlier event. Crucially, the guard is **not** a pure monotone absorbing rank (S3's lot-board model) because `Extension` is a sanctioned `Disputed → Active` *backward* transition — only the terminal states (`Fulfilled`, and the `Refund`/`Closed` terminal resolution) are absorbing. Implement the terminal-absorbing guard precisely and leave the `Disputed → Active` path open.
4. **`AddOperationsModule()` — additive registration** — contribute the `OperationsObligationsView` document type (alongside the existing `SettlementQueueView`, `LotBoardView`, `BidActivityEntry`) via `services.ConfigureMarten()` in the same `operations` schema. **No** new `AddMarten()` call; no saga, no aggregate, no local event-stream registration (documents only — the S2/S3 "pure Path A consumer needs no event-graph registration" learning holds). Add any Marten index the view's query axes warrant (e.g. on `QueueState` and/or `ListingId`); record the choice in the retro.
5. **`Program.cs` wiring (routing-only)** — add the `operations-obligations-events` `ListenToRabbitQueue()` consumer; add publish-route additions for the **four S4-consumed events** (`DeadlineEscalated`, `DisputeOpened`, `DisputeResolved`, `ObligationFulfilled`) to `operations-obligations-events`; `AutoProvision()` declares the new queue. These events currently route only to `relay-obligations-events`; S4 adds the **parallel** `operations-obligations-events` publish routes — all four are Obligations-published on the same source family, so there is **no** cross-queue asymmetry like S3's `ListingWithdrawn` (confirm this at session start). **No upstream Obligations BC code change** — these are routing additions to existing message types.

## Conventions to pin or follow

- **Read-model strategy — ADR 014 Path A, Sub-Option A.** One view per logical entity (`ObligationId`), one sibling handler family per source BC (here exactly one — Obligations), tolerant upsert, status-preservation guards. `docs/skills/marten-projections/SKILL.md` owns the mechanics; the prompt points.
- **Upsert, not append.** The obligations view is an **upsert** view (one row per `ObligationId`, mutated in place across the four events). It is **not** an append/feed like the bid-activity feed.
- **Non-monotone state machine.** Unlike the lot board's monotone absorbing rank (S3), this view has a sanctioned backward transition (`Disputed → Active` on `Extension`). Only the terminal states (`Fulfilled`; `Refund`/`Closed` resolutions) are absorbing — guard those, leave `Disputed → Active` open. Do not force-fit the S3 rank.
- **Operations is a pure consumer.** Consumer handlers return `Task` and write via the injected Marten session — they do **not** return `OutgoingMessages` and make **no** `IMessageBus` calls. `bounded-contexts.md` §Operations "Integration out: None" is enforced structurally; assert it in a test (the S2/S3 `tracked.Sent` precedent).
- **No-default-scheme constraint.** S4 introduces no authorization attribute and registers no auth scheme; it leaves `Program.cs`'s auth state untouched. Operations has no HTTP endpoints in S4.
- **Module shape** per `docs/skills/adding-bc-module/SKILL.md`: additive `services.ConfigureMarten()` inside `AddOperationsModule()`; no `AddMarten()` inside the module; same `operations` schema as S2/S3.
- **Canonical bootstrap + `AutoApplyTransactions`** stay in `Program.cs`; the module only contributes the type and the routes.
- **Cross-BC fixture exclusion naming:** reuse the existing per-project `OperationsBcDiscoveryExclusion` pattern per `critter-stack-testing-patterns/SKILL.md`.
- `sealed record`; `IReadOnlyList<T>` for any collection; no "Event" suffix on type names; participant identifiers are `BidderId`/`RaisedBy`/`WinnerId`, never "paddle".

## Open questions

- **One view vs two queues vs two views.** W006 §4 freezes a **single** `ObligationId`-keyed view with derived queue membership (escalation = `Escalated`, open-dispute = `Disputed`). Default to that single-view-with-`QueueState`-discriminator shape. Do **not** split into two physical views or a queue-discriminator-row model without amending W006 first — if the query ergonomics of the two derived queues seem to demand a split, stop and flag rather than re-deciding the freeze.
- **`QueueState` member set and the two terminal flavours.** W006 §4's derivation reads `Escalated → Disputed → (Resolved or back to active) → Fulfilled`. The "back to active" (`Extension`) outcome and the terminal `Refund`/`Closed` "resolved" outcome are **two distinct** results of `DisputeResolved`, and `Fulfilled` is a third terminal. The exact member set — e.g. whether `Active` (post-`Extension`) and a terminal `Resolved` (post-`Refund`/`Closed`) are distinct members, and whether `Fulfilled` and the terminal `Resolved` collapse — is an implementation choice W006 leaves unfrozen. Pin a member set that faithfully realises the derivation and keep it Operations-internal (no Contracts type, consistent with `LotBoardStatus`/`SettlementQueueStatus`); flag the chosen mapping in the retro.
- **Row-removal vs status-flip on resolve/fulfil.** W006 §4 says `DisputeResolved`/`ObligationFulfilled` *"drop the row from every active queue"* — this is membership-by-`QueueState`-filter, **not** a Marten document delete. Default to a **status-flip** (the row persists with a terminal/active `QueueState`; the queues are query filters), preserving the obligation's history for the operator. Do **not** physically delete the document on resolve/fulfil without flagging — confirm at session start that "drop from queue" means "filter out," not "remove row."
- **`DisputeResolved`/`ObligationFulfilled` for an unseen obligation.** A happy-path obligation can fulfil (narrative 006) **without** ever escalating or disputing — so `ObligationFulfilled` (or a stray `DisputeResolved`) may arrive for an `ObligationId` this view has never seen. Decide and pin: is it a **no-op** (this view only tracks obligations that entered an escalation/dispute queue) or a **tolerant-upsert seed** (a terminal `Fulfilled` row materialises with the fulfilment fields and nulls elsewhere)? W006 §4's tolerant-upsert default leans toward seed, but a view whose stated purpose is the escalation/dispute queues may prefer no-op. This is a genuine design decision — flag it rather than guessing.
- **Out-of-order / re-delivery guard on the non-monotone path.** Because `Extension` is a sanctioned `Disputed → Active` backward transition, a naive monotone absorbing rank (S3) cannot guard this view. Confirm the guard distinguishes *terminal* absorption (`Fulfilled`, `Refund`/`Closed`) — which must resist any earlier-event re-delivery — from the *legitimate* `Disputed → Active` move. If out-of-order delivery (e.g. a late `DeadlineEscalated` after `DisputeOpened`) could regress `Disputed → Escalated`, decide whether a per-transition ordering guard or an event-timestamp comparison is needed, and flag the choice.
