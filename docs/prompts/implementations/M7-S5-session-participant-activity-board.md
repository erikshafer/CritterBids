## Goal

Land the **fifth** Operations read model(s) on top of the M7-S2/S3/S4 scaffold: the **session activity board** and the **participant activity board** — the two W006 §5a/§5b operator surfaces. Both are **pure-consumer** ADR 014 Path A **upsert** views fed off RabbitMQ, one row per logical entity (`SessionId` / `ParticipantId`), mutated in place. This slice routes the **three Auctions session events** onto the **existing** `operations-auctions-events` queue (S3 deliberately left them unrouted) and wires **one new consumer queue** (`operations-participants-events`) for `ParticipantSessionStarted`; it implements the two views with one handler family per source BC; and it proves both **end-to-end** with Testcontainers projection tests against a real Postgres. The M7-S2/S3/S4 BC module, the `operations` schema, the Testcontainers + Alba fixture, and the per-project `OperationsBcDiscoveryExclusion` pattern already exist — **build on them, do not recreate them**.

W006 §5a/§5b froze both view field sets before this slice opened; every field already traces to an existing `CritterBids.Contracts.Auctions.*` / `CritterBids.Contracts.Participants.*` payload (or is a marked *derived* entry — the session `Status`). Implement those fields — no more, no fewer. If a new design decision surfaces mid-session, stop and flag (see Open Questions) — do not pivot in-session.

## In scope

The four consumed integration events (already in `CritterBids.Contracts`, no Contracts change) and their exact payloads — implement against these field names and types verbatim:

- **`SessionCreated`** (Auctions) — `SessionId` (Guid), `Title` (string), `DurationMinutes` (int), `CreatedAt` (DateTimeOffset)
- **`SessionStarted`** (Auctions) — `SessionId` (Guid), `ListingIds` (`IReadOnlyList<Guid>`), `StartedAt` (DateTimeOffset)
- **`ListingAttachedToSession`** (Auctions) — `SessionId` (Guid), `ListingId` (Guid), `AttachedAt` (DateTimeOffset)
- **`ParticipantSessionStarted`** (Participants) — `ParticipantId` (Guid), `DisplayName` (string), `BidderId` (**string**), `CreditCeiling` (decimal), `StartedAt` (DateTimeOffset)

1. **Session activity board view** — a `sealed record` Marten document keyed by `SessionId`, carrying exactly the W006 §5a field set: `SessionId`, `Title` (`string`, from `SessionCreated`), `DurationMinutes` (`int`, from `SessionCreated`), `AttachedListingIds` (`IReadOnlyList<Guid>`), `Status` (a *derived* enum, monotone `Created → Started`, set from the event type — not from a payload field), `CreatedAt` (`DateTimeOffset`, from `SessionCreated`), `StartedAt` (`DateTimeOffset?`, from `SessionStarted`, null until started). The enum's member spelling and type name are an implementation choice (W006 freezes the derivation, not the names), kept Operations-internal (no Contracts type, consistent with `LotBoardStatus`). **No `LastUpdatedAt`** — it is not in the freeze (the S4 precedent: no extra fields).
2. **Participant activity board view** — a `sealed record` Marten document keyed by `ParticipantId`, carrying exactly the W006 §5b field set: `ParticipantId`, `DisplayName` (`string`), `BidderId` (**`string`** on the payload — not `Guid`; never "paddle"), `CreditCeiling` (`decimal`), `StartedAt` (`DateTimeOffset`). All five fields come from the single `ParticipantSessionStarted` payload. **No `LastUpdatedAt`**.
3. **One Auctions-source session-activity handler family — ADR 014 Sub-Option A, a single sibling class** (Auctions is the only source BC for the session board) consuming the three session events, each `Handle` a tolerant upsert (load-or-construct by `SessionId`, mutate via record `with`, `Store`), pure-consumer (returns `Task`, **no** `OutgoingMessages`, **no** `IMessageBus`):
   - **`SessionCreated`** → seed the row in `Status = Created`; set `Title`/`DurationMinutes`/`CreatedAt`.
   - **`SessionStarted`** → advance `Status` to `Started`; set `StartedAt`; **union** `ListingIds` into `AttachedListingIds` (dedupe).
   - **`ListingAttachedToSession`** → **union** the single `ListingId` into `AttachedListingIds` (dedupe); does **not** change `Status`.
4. **One Participants-source participant handler** — a single sibling handler consuming `ParticipantSessionStarted` as a tolerant upsert over the `ParticipantId`-keyed document, populating all five fields. Pure-consumer (returns `Task`, no `OutgoingMessages`, no `IMessageBus`).
5. **`AttachedListingIds` union-and-dedupe semantics** — the field is an **additive set-union**, not a last-write replace: `ListingAttachedToSession` ids accumulate across deliveries, and `SessionStarted.ListingIds` is unioned (not assigned over) the accumulated set, with duplicates removed. A `ListingAttachedToSession` that arrives **after** `SessionStarted` (out-of-order) must add its id to the set **without** regressing `Status` from `Started` back to `Created`. Implement the union precisely; do not replace-on-`SessionStarted`.
6. **Status-preservation guard (session board)** — `Status` is a monotone `Created → Started` rank (the S3 lot-board absorbing-rank pattern applies cleanly here — there is no sanctioned backward transition, unlike S4): a re-delivered `SessionCreated` after `SessionStarted` must **not** regress `Status` to `Created` or null `StartedAt`. The participant board has a single event and needs no status guard, only idempotent-upsert tolerance.
7. **`AddOperationsModule()` — additive registration** — contribute the two new view document types (alongside the existing `SettlementQueueView`, `LotBoardView`, `BidActivityEntry`, `OperationsObligationsView`) via `services.ConfigureMarten()` in the same `operations` schema. **No** new `AddMarten()` call; no saga, no aggregate, no local event-stream registration (documents only — the S2/S3/S4 "pure Path A consumer needs no event-graph registration" learning holds). Add any Marten index the views' query axes warrant; record the choice in the retro (see Open Questions — the session-board index set given S6's query needs is genuinely undecided).
8. **`Program.cs` wiring (routing-only)** —
   - **Existing `operations-auctions-events` queue:** add the three **session-event publish routes** (`SessionCreated`, `SessionStarted`, `ListingAttachedToSession`) to the **already-declared** `operations-auctions-events` queue. S3 deliberately left these unrouted — see the comment at `Program.cs` ~L202–204. **Reuse S3's existing `ListenTo`** for that queue; add **no** new auctions listener and no new auctions queue.
   - **New `operations-participants-events` queue:** add the `ParticipantSessionStarted` publish route, a new `ListenToRabbitQueue("operations-participants-events")`, and let `AutoProvision()` declare it at startup.
   - **No upstream Auctions / Participants BC code change** — these are routing additions to existing message types only.
9. **Cross-BC discovery exclusions** — extend the existing per-project `OperationsBcDiscoveryExclusion` pattern to every additional existing BC test fixture whose in-process host now breaks because it discovers the new Operations session/participant handlers and the four S5 events flip to sticky `Separated` routing. **Find the exact set by a red full-suite run** — do **not** predict it in code (S3/S4 predictions over-shot; the red run is the only source of truth). Apply the `SendMessageAndWaitAsync`-not-`Invoke` rule to any foreign-fixture dispatch of a now-fanned-out S5 event.
10. **`CritterBids.Operations.Tests` additions** — reuse the existing Testcontainers + Alba fixture. Add **at least one real-Postgres projection test per view** (milestone §1 exit criterion: "at least one read-model projection test per operator view exercising the consume → upsert/append path"):
    - **session board** — `SessionCreated` seeds a row in `Status == Created`; one or more `ListingAttachedToSession` accumulate ids; `SessionStarted` advances `Status → Started`, sets `StartedAt`, and **unions** its `ListingIds` with the already-attached ids (dedupe asserted); an out-of-order `ListingAttachedToSession` **after** `SessionStarted` adds its id **without** regressing `Status`;
    - **participant board** — `ParticipantSessionStarted` seeds a row with all five fields, asserting `BidderId` round-trips as a **`string`**;
    - **pure-consumer assertion** — a `tracked.Sent`-empty assertion proving both handler families publish nothing (the S2/S3/S4 precedent);
    - **schema-mapping assertion** — a one-line `information_schema` check that both views map to the `operations` schema (the S2 learning that behaviour tests are schema-blind).

## Conventions to pin or follow

- **Read-model strategy — ADR 014 Path A, Sub-Option A.** One view per logical entity (`SessionId`, `ParticipantId`), one sibling handler family per source BC (here exactly one each — Auctions for the session board, Participants for the participant board), tolerant upsert, status-preservation guards. `docs/skills/marten-projections/SKILL.md` owns the mechanics; the prompt points.
- **One view per entity (ADR 014).** Two distinct views, two distinct keys — **not** a combined session+participant document. The session board keys on `SessionId`; the participant board keys on `ParticipantId`.
- **Upsert, not append.** Both views are **upsert** (one row per entity, mutated in place). Neither is an append/feed like the bid-activity feed.
- **Set-union accumulation, not replace.** `AttachedListingIds` is additive set-union with dedupe across `ListingAttachedToSession` and `SessionStarted.ListingIds` — never a last-write assignment. `IReadOnlyList<Guid>` per the global record convention.
- **Monotone status (session board).** `Created → Started` is a forward-only rank (the S3 absorbing-rank pattern applies cleanly — no backward transition, unlike S4's `Extension` path). Guard against regression on re-delivery.
- **Operations is a pure consumer.** Consumer handlers return `Task` and write via the injected Marten session — they do **not** return `OutgoingMessages` and make **no** `IMessageBus` calls. `bounded-contexts.md` §Operations "Integration out: None" is enforced structurally; assert it in a test (the S2/S3/S4 `tracked.Sent` precedent).
- **No-default-scheme constraint.** S5 introduces no authorization attribute and registers no auth scheme; it leaves `Program.cs`'s auth state untouched. Operations has no HTTP endpoints in S5.
- **Module shape** per `docs/skills/adding-bc-module/SKILL.md`: additive `services.ConfigureMarten()` inside `AddOperationsModule()`; no `AddMarten()` inside the module; same `operations` schema as S2/S3/S4.
- **Canonical bootstrap + `AutoApplyTransactions`** stay in `Program.cs`; `Program.cs` contributes the routes and `AddOperationsModule()` only contributes the document registrations and indexes. UUID v7 stream IDs are N/A this slice — both views are keyed by inbound ids (`SessionId`/`ParticipantId`), and Operations creates no streams.
- **Cross-BC fixture exclusion naming:** reuse the existing per-project `OperationsBcDiscoveryExclusion` pattern per `critter-stack-testing-patterns/SKILL.md`; find the set by a red full-suite run.
- `sealed record`; `IReadOnlyList<T>` for any collection; no "Event" suffix on type names; participant identifier is `BidderId` (a `string` here), never "paddle".
