## Deliverable 5: `[WriteAggregate]` stream-ID verification

This is the one deliverable that may produce a code change. Address the fragility documented in
`docs/retrospectives/M2-listings-pipeline-retrospective.md` under "What M3 Should Know."

### Background

`SubmitListingHandler.Handle()` carries this signature:

```csharp
public static (Events, OutgoingMessages) Handle(
    SubmitListing cmd,
    [WriteAggregate] SellerListing listing)
```

The `SubmitListing` command is:

```csharp
public sealed record SubmitListing(Guid ListingId, Guid SellerId);
```

Wolverine's `[WriteAggregate]` attribute instructs the framework to load the event-sourced
aggregate from Marten before calling the handler. To do this, Wolverine must resolve the stream ID
from the incoming command. By convention, for aggregate type `SellerListing`, Wolverine looks for a
property named `SellerListingId` on the command. The `SubmitListing` command has `ListingId` —
**not** `SellerListingId`.

The existing tests in `SubmitListingTests.cs` call `SubmitListingHandler.Handle(cmd, listing)`
directly with a pre-built `SellerListing` instance. They bypass Wolverine's dispatch path entirely.
The `[WriteAggregate]` convention has never been exercised.

### Step 1 — Consult the skill and library docs

Before writing any code, check `docs/skills/wolverine-message-handlers.md` for any existing
documentation on the `[WriteAggregate]` naming convention or the `AggregateIdMember` parameter.

Then use Context7 (Wolverine library ID: `/jasperfx/wolverine`) to look up `[WriteAggregate]`
documentation. Specifically confirm:

- The exact property name convention Wolverine uses to resolve the stream ID from the command
- Whether `[WriteAggregate]` accepts an explicit member name override (e.g., `AggregateIdMember`)
- Whether there is a fallback (e.g., a property simply named `Id`)

Document what you find in the retrospective regardless of outcome.

### Step 2 — Write the verification test

Create `tests/CritterBids.Selling.Tests/SubmitListingDispatchTests.cs`.

This test class exercises `SubmitListing` through **the full Wolverine dispatch path**, not by
calling Handle directly. It must:

1. Spin up the full application host with Testcontainers PostgreSQL
2. Disable all external Wolverine transports (`DisableAllExternalWolverineTransports()`)
3. Create a valid `SellerListing` stream by sending `CreateDraftListing` through the API or by
   seeding the event stream directly via `IDocumentSession` — use whichever approach the skill
   file recommends for aggregate setup in integration tests
4. Ensure a `RegisteredSeller` row exists for the seller (seed via `IDocumentSession` directly
   against the `selling` schema, or reuse the pattern from `SubmitListingTests` fixtures)
5. Invoke `SubmitListing` via `IMessageBus.InvokeAsync()` — this is the key step that exercises
   the full Wolverine dispatch path including `[WriteAggregate]` stream ID resolution
6. Assert the listing is now in Published state: load via
   `IQuerySession.Events.AggregateStreamAsync<SellerListing>(listingId)` and assert
   `listing.Status == ListingStatus.Published`

Test method name:
```csharp
[Fact]
public async Task SubmitListing_ViaWolverineDispatch_LoadsAggregateAndProducesPublishedListing()
```

Follow the fixture and test setup patterns in `docs/skills/critter-stack-testing-patterns.md`
exactly. Reuse the existing `SellingTestFixture` if it is compatible with `IMessageBus` injection;
otherwise create a `SubmitListingDispatchFixture` alongside it.

### Step 3 — Interpret the result and apply a fix if needed

**If the test passes without any change:** The `[WriteAggregate]` convention works — either
Wolverine has a fallback that matches `ListingId`, or the skill/library docs reveal why it works.
Document the finding. No code change needed.

**If the test fails with a null aggregate or stream-not-found error:** The naming mismatch is
confirmed. Apply the fix and add a skill update.

#### Fix to apply (if needed)

**Option A — Explicit attribute parameter (preferred):**

In `SubmitListing.cs`, change `[WriteAggregate]` to specify the member name explicitly:

```csharp
public static (Events, OutgoingMessages) Handle(
    SubmitListing cmd,
    [WriteAggregate(AggregateIdMember = nameof(SubmitListing.ListingId))] SellerListing listing)
```

This is the least invasive change. The command shape, all existing tests, and the `SubmitListing`
record remain unchanged.

**Option B — Rename the command property:**

Rename `SubmitListing.ListingId` to `SubmitListing.SellerListingId` to match Wolverine's
conventional expectation. Update all test references to `cmd.ListingId` accordingly.

Prefer Option A unless the skill file or library docs explicitly recommend Option B as the
idiomatic approach.

**Option C — Apply Option A and document the pattern:**

If Option A is applied, also add a note to `docs/skills/wolverine-message-handlers.md`
documenting the anti-pattern and the fix:

> **`[WriteAggregate]` — stream ID property naming.** Wolverine resolves the stream ID from
> the command using a property named `{AggregateTypeName}Id`. When the command uses a different
> property name (e.g., `ListingId` for a `SellerListing` aggregate), specify the member
> explicitly: `[WriteAggregate(AggregateIdMember = nameof(SomeCommand.ListingId))]`.
> Without this, dispatch fails silently with a null aggregate at runtime.

Add this note to the appropriate anti-pattern or convention section in the skill file.

Check `src/CritterBids.Selling/UpdateDraftListing.cs` to confirm whether it uses `[WriteAggregate]`
