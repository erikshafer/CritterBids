## D5: `[WriteAggregate]` finding

### What the skill file and Context7 docs say

`docs/skills/wolverine-message-handlers.md` documented the conventional resolution
(`{AggregateTypeName}Id`) but did **not** describe the override mechanism. The session prompt
suggested `[WriteAggregate(AggregateIdMember = nameof(SubmitListing.ListingId))]` as a named
property — this syntax does not compile.

`WriteAggregateAttribute` source (WebFetch against
`github.com/JasperFx/wolverine/blob/main/src/Persistence/Wolverine.Marten/WriteAggregateAttribute.cs`)
shows the attribute is a **positional** constructor parameter:

```csharp
public WriteAggregateAttribute(string? routeOrParameterName = null)
```

No `AggregateIdMember` property exists. The correct invocation is
`[WriteAggregate("ListingId")]` or, refactor-safely, `[WriteAggregate(nameof(SubmitListing.ListingId))]`.

The attribute resolves the stream ID by looking, in order, for:

1. The string passed to the constructor — if present, used as-is (route parameter or command property)
2. A command property named `{AggregateTypeName}Id` (e.g., `SellerListingId` for `SellerListing`)
3. A command property named `Id`

Resolution happens at **handler code-gen time**, which runs lazily at first dispatch. That is why
the existing 30 direct-call tests never caught the `ListingId` vs `SellerListingId` mismatch — they
never went through Wolverine.

### Whether the dispatch test passed without changes

**It did not.** First run failed with:

```
System.InvalidOperationException : Unable to determine an aggregate id for the parameter 'listing'
on method HandlerType: CritterBids.Selling.SubmitListingHandler,
Method: System.ValueTuple`2[Wolverine.Marten.Events,Wolverine.OutgoingMessages]
  Handle(CritterBids.Selling.SubmitListing, CritterBids.Selling.SellerListing)
   at Wolverine.Marten.WriteAggregateAttribute.Modify(IChain chain, ParameterInfo parameter,
      IServiceContainer container, GenerationRules rules)
```

Confirmed the fragility called out in the M2 retrospective.

### Fix applied

Option A — explicit override with `nameof` for refactor safety.

```csharp
public static (Events, OutgoingMessages) Handle(
    SubmitListing cmd,
    [WriteAggregate(nameof(SubmitListing.ListingId))] SellerListing listing)
```

Option B (rename `ListingId` → `SellerListingId` on the command) was rejected because:

- `ListingId` is the name a caller expects — `SellerListing` is a BC-internal aggregate name
- Contract events (`ListingPublished` in `CritterBids.Contracts.Selling`) already use `ListingId`
- The `nameof` arg keeps the rename-safety of Option B without the naming dissonance

### Final state of `SubmitListingHandler.Handle()`

```csharp
public static (Events, OutgoingMessages) Handle(
    SubmitListing cmd,
    [WriteAggregate(nameof(SubmitListing.ListingId))] SellerListing listing)
```

The command shape and all 30 direct-call tests remain unchanged.

## Key learnings

1. **Wolverine code-gen failures surface at first dispatch, not at startup.** An aggregate handler
   with a misnamed stream-ID property builds clean, passes every direct-call test, registers at
   startup, and detonates only when a message actually runs through `IMessageBus`. Any BC that
   accepts commands through `[WriteAggregate]` needs at least one end-to-end dispatch test per
   command shape, not just direct-call unit tests. Plan one per new command in M3.

2. **`[WriteAggregate]` override is a positional constructor arg, not a named property.** The
   attribute's public API is `WriteAggregateAttribute(string? routeOrParameterName)`. Several
   sources (including the prompt authored for this session) referenced a nonexistent
   `AggregateIdMember` property. Confirm attribute shapes against the Wolverine source when the
   name is not obviously canonical. `docs/skills/wolverine-message-handlers.md` now captures this.

3. **Documentation drift accumulates silently when the ground truth moves faster than the index.**
   ADR 011 landed in M2-S4; the Participants migration landed in M2-S5; but `CLAUDE.md`, the MVP
   milestone doc, and `bounded-contexts.md` all carried pre-pivot language until this session. The
   retro-and-session-close discipline was not sufficient on its own — a dedicated alignment session
   was needed. For future architecture-level ADRs (011-class), treat a documentation sweep as a
   mandatory final task in the session that lands the ADR, not a separate cleanup pass.

4. **Refactor-safe attribute arguments via `nameof`.** Using
   `[WriteAggregate(nameof(SubmitListing.ListingId))]` rather than a string literal means renaming
   the command property triggers a compile error that points directly at the attribute. String
   literal arguments to framework attributes are a silent failure mode — prefer `nameof` for any
   such override across the codebase (same principle applies to `[Tags(...)]`, `[Handler(...)]`,
   MediatR-style attributes).
