## Prompt premise clarification

The prompt's acceptance criterion stated:

> "a one-line confirmation that the implicit-convention `[WriteAggregate]` count in the codebase
> is now zero"

Literally, this count is **1**, not 0, after this session. The remaining usage is
`src/CritterBids.Participants/Features/RegisterAsSeller/RegisterAsSeller.cs:54`:

```csharp
public static (IResult, SellerRegistered, OutgoingMessages) Handle(
    RegisterAsSeller cmd,
    [WriteAggregate] Participant participant)
```

This usage is **not fragile**, for two independent reasons:

1. The command property `RegisterAsSeller.ParticipantId` matches the convention
   `{AggregateName}Id` = `ParticipantId` for aggregate `Participant`. Wolverine's
   `FindIdentity` path resolves correctly without an override.
2. The handler is an HTTP endpoint (`[WolverinePost("/api/participants/{id}/register-seller")]`).
   Wolverine additionally supports resolution from the `{id}` route segment — the file's own
   docstring at `RegisterAsSeller.cs:13–17` documents this: "FindIdentity tries 'participantId'
   first, then falls back to 'id' (the route variable)."

So the prompt's premise is met in the sense that matters — **zero fragile implicit-convention
usages remain** — but the literal count of `[WriteAggregate]` attributes without arguments is 1,
not 0. Recording the nuance here rather than silently fudging the number. The `RegisterAsSeller`
handler was not modified this session per the prompt's "Do not sweep speculatively" rule.

## Key learnings

1. **Implicit `[WriteAggregate]` is fine when the command property is `{AggregateName}Id`.**
   The fragility is specifically the *divergence* case — when the aggregate type name carries a
   BC-role prefix (e.g., `SellerListing`) but the command property uses the bare domain term
   (e.g., `ListingId`). Future BCs can continue to use the implicit form when the naming aligns;
   the explicit override is only load-bearing when names diverge. `RegisterAsSeller.ParticipantId`
   → `Participant` is a working example of the former; `UpdateDraftListing.ListingId` →
   `SellerListing` was a working example of the latter once dispatch was exercised.

2. **HTTP endpoints have a second `[WriteAggregate]` resolution path.** Via `{id}` route segments,
   Wolverine can resolve stream IDs without relying on command property naming at all. This
   explains why the S2 investigation needed to distinguish "implicit-convention usage" from
   "fragile usage" — HTTP handlers with a route-based `{id}` neutralise the naming concern even
   when the command property name doesn't match. Pure command handlers (no HTTP route) only get
   the property-name path, which is why `SubmitListing` and `UpdateDraftListing` were fragile.
